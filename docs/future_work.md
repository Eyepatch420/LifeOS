# Future Work (explicitly deferred, not forgotten)

- **riverpod_lint / custom_lint** — not enabled. At time of writing, `riverpod_lint`/`custom_lint` pin an `analyzer_plugin` version incompatible with this Flutter SDK's bundled `analyzer` (7.6.0), which broke `build_runner` entirely. The project runs on `flutter_riverpod ^3.0.0` + `riverpod_annotation`/`riverpod_generator ^4.0.0` (upgraded from the originally-planned 2.x line specifically to resolve this), but `riverpod_lint`/`custom_lint` still weren't re-added since they add no functional capability, only extra static analysis. Revisit once a compatible `riverpod_lint` release targeting `riverpod_annotation` 4.x lands.
- **Rive splash animation** — no `.riv` assets exist yet. Splash currently uses code-driven `flutter_animate` (fade/scale/gradient reveal). Add `rive` package + assets when available.
- **Drift tables** — Module 1 has no relational data, so `core/database/app_database.dart` is scaffolded empty. Real tables (Reminders, Notes, Expenses, Health entries, etc.) start in Module 2.
- **Localization (`l10n/`)** — not scaffolded; no localization requirement yet.
- **Dark-mode SVG variant for the Home hero** — currently handled via a scrim overlay over the single light-mode SVG. Revisit with a proper dark-authored SVG if the scrim reads as muddy once visually tested.
- **Settings module** — `ThemeModeSelector`, `AccentColorSelector`, and toggle patterns from User Setup are built generically enough to be reused by a future Settings screen, but no such screen exists yet.
- **Home hero particle/sparkle overlay** — stretch-goal polish, only implemented if it doesn't risk dropping below 60fps.
- **Spare Lottie asset** — `assets/lottie/spare_presentation_activity.json` is not used in Module 1's 3-page onboarding; reserved for a potential 4th onboarding page or a future analytics/activity screen.
- **Reminders / Health / Finance / Documents modules** — Module 1 only has placeholder stub screens for these tabs.
- **Home Workspace settings UI** — `homeSectionsProvider` (order + visibility per section) and the rich `HomeSectionMeta` metadata (`displayName`/`icon`/`supportsCustomization`) exist specifically so a future Settings screen can render a section-toggle/reorder checklist by reading/writing this one provider, with zero duplicated metadata. No such screen exists yet.
- **`PressableScale` → `AppMotionPresets.button` retrofit** — `PressableScale` (a shared low-level primitive used by 3+ non-Home consumers) still inlines raw `AppDurations.fast`/`AppCurves.easeOutCubic` rather than referencing the named preset catalogue. Deferred since it's shared infrastructure outside any one module's scope, not a Home-specific change.
- **Real per-section data sources (Module 3+)** — each Home section's `AsyncNotifier.build()` currently just resolves a mock const. Swapping in a real repository call (Reminders, Notes, Expenses, Habits, etc.) is purely a `build()`-body change per the `AsyncNotifier` architecture — no provider/widget API changes needed.
- **Un-pairing the combined Up Next/Habit Streaks and Recent Notes/My Lists registry entries** — currently registered as one `HomeSectionMeta` each so their tablet side-by-side `ResponsiveBuilder` layout survives reordering as a unit. Splitting them into 4 independently reorderable sections is deferred until a real settings UI and design calls for it.
