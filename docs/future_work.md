# Future Work (explicitly deferred, not forgotten)

- **riverpod_lint / custom_lint** — not enabled. At time of writing, `riverpod_lint`/`custom_lint` pin an `analyzer_plugin` version incompatible with this Flutter SDK's bundled `analyzer` (7.6.0), which broke `build_runner` entirely. The project runs on `flutter_riverpod ^3.0.0` + `riverpod_annotation`/`riverpod_generator ^4.0.0` (upgraded from the originally-planned 2.x line specifically to resolve this), but `riverpod_lint`/`custom_lint` still weren't re-added since they add no functional capability, only extra static analysis. Revisit once a compatible `riverpod_lint` release targeting `riverpod_annotation` 4.x lands.
- **Rive splash animation** — no `.riv` assets exist yet. Splash currently uses code-driven `flutter_animate` (fade/scale/gradient reveal). Add `rive` package + assets when available.
- **Drift tables** — added in Module 4 Phase 1 (`Notes`/`Reminders`/`Expenses`/`Habits`+`HabitCompletions`/`Lists`+`ListItems`/`MoodEntries`/`Notifications`/`FocusSessions`, schema v2; `Reminders` gained `recurrence`/`customRule` in schema v3, Module 4 Phase 3). `Notes`/`Lists`/`Reminders` now have real repositories (Module 4 Phases 2–3); `Expenses`/`Habits`+`HabitCompletions`/`MoodEntries`/`FocusSessions` still have none — start in Phases 4+. `Notifications` now has a real writer (`NotificationEngine`, Phase 3).
- **Localization (`l10n/`)** — not scaffolded; no localization requirement yet.
- **Dark-mode SVG variant for the Home hero** — currently handled via a scrim overlay over the single light-mode SVG. Revisit with a proper dark-authored SVG if the scrim reads as muddy once visually tested.
- **Settings module** — `ThemeModeSelector`, `AccentColorSelector`, and toggle patterns from User Setup are built generically enough to be reused by a future Settings screen, but no such screen exists yet.
- **Home hero particle/sparkle overlay** — stretch-goal polish, only implemented if it doesn't risk dropping below 60fps.
- **Spare Lottie asset** — `assets/lottie/spare_presentation_activity.json` is not used in Module 1's 3-page onboarding; reserved for a potential 4th onboarding page or a future analytics/activity screen.
- **Health / Finance / Documents modules** — Module 1 only has placeholder stub screens for these tabs. Reminders is real as of Module 4 Phase 3 (`RemindersListScreen`/`ReminderDetailScreen` replace its placeholder).
- **Home Workspace settings UI** — `homeSectionsProvider` (order + visibility per section) and the rich `HomeSectionMeta` metadata (`displayName`/`icon`/`supportsCustomization`) exist specifically so a future Settings screen can render a section-toggle/reorder checklist by reading/writing this one provider, with zero duplicated metadata. No such screen exists yet.
- **`PressableScale` → `AppMotionPresets.button` retrofit** — `PressableScale` (a shared low-level primitive used by 3+ non-Home consumers) still inlines raw `AppDurations.fast`/`AppCurves.easeOutCubic` rather than referencing the named preset catalogue. Deferred since it's shared infrastructure outside any one module's scope, not a Home-specific change.
- **Real per-section data sources (Module 3+)** — each Home section's `AsyncNotifier.build()` currently just resolves a mock const. `recentNotesProvider`/`myListsProvider` (Phase 2) and `upNextProvider`/`timelineProvider` (Phase 3, via `agendaRegistryProvider`) are now real; `overviewStatsProvider`/`habitStreaksProvider` still resolve mock consts pending Habits/Mood/Finance (Phases 4+). Swapping in a real repository call is purely a `build()`-body change per the `AsyncNotifier` architecture — no provider/widget API changes needed.
- **Un-pairing the combined Up Next/Habit Streaks and Recent Notes/My Lists registry entries** — currently registered as one `HomeSectionMeta` each so their tablet side-by-side `ResponsiveBuilder` layout survives reordering as a unit. Splitting them into 4 independently reorderable sections is deferred until a real settings UI and design calls for it.
- **New Document quick action** — `kQuickActions`'s `new_document` entry stays a safe no-op (falls through `QuickActionsRow`'s existing `?.call()` guard); Documents is its own future module and a document-creation flow belongs there, not duplicated in Home's push-navigation pass.
- **Motivational banner tap destination** — `HomeHeroSection.onBannerTap` stays unwired; no destination screen is defined anywhere in the Module 3 scope for it.
- **Quick Action screens (New Note/Reminder/Expense/Habit) stay mock-backed** — each will get its own tiny `AsyncNotifier<List<Create*Request>>` provider (Phase 2), mirroring `home_providers.dart`'s existing mock pattern rather than standing up real Drift-backed repositories now. Swapping in real persistence later is a `build()`-body-only change per the same seam already promised for the 7 dashboard sections above.
- **`HomeSectionMeta.enabled`/`.collapsed`/`.priority`** — added in Module 3 Phase 1 as pure data-shape reservations (same treatment as `analyticsName`/`supportsCustomization`); no rendering code reads them yet. Intended for a future Settings/personalization screen.
- **Search result taps for upNext/reminder categories** — still `pop()` back to Home; note/list categories now navigate to real detail screens (Module 4 Phase 2D). Reminders gains its own once Phase 3 lands.
- **Search index — note/list contributors now real** (Module 4 Phase 2A/2B/2C: `NotesSearchContributor`/`ListsSearchContributor` registered in `config/di/search_contributor_registrations.dart`). Health/Finance/Documents/Settings tabs and the Up Next category are still fixed placeholder entries in `search_providers.dart` — each becomes a real contributor as its feature lands.
- **`core/services` scaffolds are no-ops** (`BackgroundService`, `NotificationScheduler`, `SyncManager`, `ForegroundTimer` — added Module 4 Phase 1) — real implementations (`flutter_local_notifications` + `timezone` for scheduled notifications, a true foreground service for Focus, remote sync) are a Module 4 Phase 6 planning deliverable (`docs/background_services_plan.md`), not implemented yet. `ClockManager` is the one exception — already real (`SystemClockManager`).
- **`FocusSessions` table exists with no repository yet** — Module 4 Phase 4 (Habits + Focus + Mood) adds the `features/focus` repository/dashboard provider/UI; the table is scaffolded in Phase 1 alongside the others purely to avoid a later schema migration.
- **NotificationContributor has no subscriber yet** — Notes/Lists each ship one (Module 4 Phase 2B/2C, proving the fifth contract piece), but the NotificationEngine that actually subscribes to the event bus and persists a feed lands in Module 4 Phase 3 alongside Reminders.
- **`ListsRepository.watchAll()` doesn't live-update on item-level changes** — it composes each list's items via a one-shot read (`ListsDao.getItems`), not a live stream, to avoid a `Stream.asyncMap` + nested-stream deadlock discovered under `flutter_test`'s fake-async zone (see `implemented_features.md`'s Phase 2 Testing notes). This means `ListsScreen`'s progress-ring/`"X/Y done"` subtitle won't refresh live if an item is checked from another screen without `ListsScreen` itself remounting. `ListDetailScreen`'s `watchById()` is unaffected (it composes a genuinely live item stream for the one list being viewed). Revisit if the summary view's staleness becomes a real usability issue.
- **DashboardRegistry is still named-provider-based, not contributor-based** — `dashboardSections(ref)` (Module 4 Phase 2D) restructured Home's registry into a joined `DashboardSection` list, but it still calls `ref.watch(notesDashboardProvider)`/`ref.watch(listsDashboardProvider)` by name inside `buildHomeSectionBuilders`. The longer-term `DashboardContributor` interface (mirroring `SearchContributor`/`AgendaContributor`, so Home's registry becomes pure `sort → render` with zero hardcoded provider names) is documented in `docs/architecture_principles.md` as a future refinement once 3+ features exist to prove the pattern against — not a Phase 2 requirement.
