## Module 4: Home Module Completion (Architecture v3)

Completes the Home module for real: every remaining feature (Notes, Reminders, Habits, Expenses, Lists, Mood, Focus, Notifications, Settings, Search) becomes a fully persisted, independently-owned feature under the `Feature UI → Riverpod → Repository → Feature Contract (Summary DTO) → Dashboard Provider → Home` flow, replacing every mock-backed `AsyncNotifier` and in-memory `Create*Request` notifier from Modules 2–3. See `docs/architecture_principles.md` for the governing rules (Type A/B/C classification, the five-part feature contract, the Golden Rule) and the phased roadmap in the corresponding planning session.

### Phase 1: Persistence foundation, event bus, service scaffolds

- [x] `docs/architecture_principles.md` — Type A/Type B/Type C feature classification, the six Architecture Constraints, and the LifeOS Golden Rule (no feature imports another feature's repository/DAO/entity/screen — only via dashboard summary providers, Search/Agenda contributors, Notification events, and shared services).
- [x] 9 new Drift tables (`core/database/tables/`): `Notes` (markdown body, pin, soft-delete), `Reminders` (dueAt, urgent, completed, soft-delete), `Expenses` (amount, category, soft-delete), `Habits` + `HabitCompletions` (FK cascade, `localDate` `yyyy-MM-dd` keys), `Lists` + `ListItems` (FK cascade; `parentListId`/`templateId`/`recurrence` reserved for future nesting/templates/recurrence without a migration), `MoodEntries` (one row per local day), `Notifications`, `FocusSessions`. `Lists`/`Notifications` use `@DataClassName` (`ListRecord`/`NotificationRecord`) to avoid shadowing `dart:core.List` and Flutter's `Notification` widget type. `AppDatabase.schemaVersion` bumped 1→2 with an `onUpgrade` migration creating all 9 tables; `AppMetadata` kept (not dropped) since dropping it on upgrade would be an unforced destructive step nothing benefits from. `PRAGMA foreign_keys = ON` set in `beforeOpen` so the cascade deletes on `HabitCompletions`/`ListItems` actually enforce.
- [x] 8 DAOs (`core/database/daos/`), one per feature, each exposing `Stream<List<Entity>>` watches + soft-delete/restore/upsert mutations following the same shape: `NotesDao`, `RemindersDao`, `ExpensesDao`, `HabitsDao` (habit + completions, `toggleCompletion(habitId, localDate, idFactory)`), `ListsDao` (lists + items, `reorderItems` via a batched write), `MoodEntriesDao` (`upsertForDate` — overwrites rather than duplicates same-day entries), `NotificationsDao`, `FocusSessionsDao` (`watchActive` — the single in-progress session, if any).
- [x] `AppDatabase.forTesting(QueryExecutor)` constructor added so tests inject `NativeDatabase.memory()`/a real temp-file connection instead of touching the platform channel `openConnection()` requires.
- [x] `databaseProvider` (`config/di/core_providers.dart`) — the GetIt→Riverpod bridge for `AppDatabase`, following the exact pattern already used for `preferencesServiceProvider`/`secureStorageServiceProvider`/`biometricServiceProvider`. Every feature repository will depend on this (or a DAO getter on it), never on `getIt<AppDatabase>()` directly.
- [x] `core/events/` — `DomainEvent` (abstract base: `sourceModule`/`sourceId`) + `EventBus` (broadcast `Stream<DomainEvent>`) + `eventBusProvider`. Repositories will emit onto this instead of writing notifications directly; the NotificationEngine (Phase 3) is the first subscriber, with analytics/achievements/widgets as documented future consumers.
- [x] `core/services/` scaffolds — `BackgroundService`, `NotificationScheduler`, `ForegroundTimer`, `SyncManager` (all no-op implementations, registered in GetIt) + `ClockManager` (the one real implementation, `SystemClockManager` wrapping `DateTime.now()`, since `todayProvider` depends on it today). `ForegroundTimer` registered as a GetIt factory (not singleton) since each future Focus session needs its own instance.
- [x] `LocalDateKey` extension (`core/extensions/datetime_extensions.dart`) — `DateTime.localDateKey` → `yyyy-MM-dd` local time, the canonical day-boundary key every feature will use for habit completions, mood entries, and per-day agenda dismissals.
- [x] `todayProvider` (`config/di/core_providers.dart`) — today's `localDateKey`, re-evaluated once a minute via `clockManagerProvider`; the shared seam for lazy midnight rollover with no background job required. Deliberately lives in `core/di`, not `features/home`, since Home is only one of several future consumers.
- [x] Verified via `flutter analyze` (clean), `flutter test` (183 tests passing, 47 new; same single pre-existing unrelated failure as every prior module — confirmed via `git stash` against a clean checkout), `dart format` (clean on all files touched by this phase), `build_runner` (clean — 9 new tables + 8 new DAOs generated).
